# Firefeet 대규모 트래픽 대응 서버 튜닝 계획 (10,000 동접자 기준)

현재 Firefeet 대시보드는 1인 관제용 MVP(Minimum Viable Product) 상태로, 프론트엔드에서 백엔드를 5초마다 폴링(Polling)하여 CSV 파일 3개와 디렉토리 목록을 매번 하드디스크에서 읽어오는 구조입니다. 
동시 접속자가 10,000명 수준으로 증가할 경우 발생하는 치명적인 디스크 I/O 병목과 CPU 과부하를 해결하기 위해 다음과 같은 단계별 아키텍처 개선이 필요합니다.

## Phase 1: 통신 아키텍처 전환 (Polling -> WebSocket)
**목표: 불필요한 네트워크 트래픽 99% 감소 및 서버 요청 최소화**

*   **현재 문제점**: 10,000명이 5초마다 API를 폴링하면 초당 수천 건의 HTTP 요청이 발생하여 서버 리소스가 고갈됩니다 (Thundering Herd Problem).
*   **해결 방안 (웹소켓 기반 푸시 시스템 도입)**:
    1.  클라이언트(브라우저) 접속 시 `/ws/dashboard` (예시) 웹소켓 엔드포인트에 1회만 연결합니다.
    2.  백엔드는 트레이딩 봇이나 리포터 봇이 작업을 완료하여 **데이터 변동이 발생했을 때만** (Event-Driven) 웹소켓을 통해 클라이언트들에게 변경된 데이터를 Push 브로드캐스트합니다.
    3.  클라이언트는 수동 폴링 없이 새로 들어온 메시지만을 UI 상태에 반영하여 렌더링합니다.

## Phase 2: 데이터 추상화 및 인메모리 캐싱 (Disk -> RAM)
**목표: 디스크 I/O (HD/SSD 병목) 완전 제거**

*   **현재 문제점**: API 요청이 올 때마다 백엔드가 `pandas.read_csv()`를 호출하여 하드디스크의 파일을 읽고 파싱합니다. 대규모 읽기 요청 시 I/O 병목 현상과 CPU Load가 발생합니다.
*   **해결 방안 (Redis 인메모리 DB 도입)**:
    1.  서버 전역 메모리 캐시나 초고속 인메모리 데이터베이스인 **Redis**를 도입합니다.
    2.  봇이 매매 결과를 CSV에 쓰는 동시에 Redis의 `logs:scalp`, `logs:swing` 리스트에도 데이터를 `Push`하여 최신화합니다 (Write-Through 캐싱).
    3.  최초 접속자가 데이터를 요청할 때 백엔드는 디스크의 CSV 파일 대신 **Redis에 보관된 JSON화된 문자열을 0.001초만에 바로 읽어 내려보냅니다**.

## Phase 3: 인프라 로드 밸런싱 및 분산 처리 (Scale-Out)
**목표: 단일 프로세스의 한계 극복 및 무중단 고가용성 보장**

*   **현재 문제점**: Uvicorn 단일 프로세스는 비동기를 지원하더라도 10,000개의 소켓을 혼자 감당하기에는 한계에 봉착합니다.
*   **해결 방안**:
    1.  **Nginx (리버스 프록시)**: 백엔드 서버 앞단에 Nginx 웹서버를 두어 정적 파일(마크다운 리포트 등)에 대한 요청은 FastAPI까지 가지 않고 Nginx가 직접 캐시해서 뿌려주도록 세팅합니다.
    2.  **UVicorn Workers**: Gunicorn이나 Uvicorn을 이용해 Worker 프로세스를 서버의 CPU 코어 수(예: 4~8개)만큼 병렬로 띄웁니다.
    3.  **Load Balancer (Scale-Out)**: 트래픽이 더 늘어난다면 서버(EC2 인스턴스)를 여러 대 띄우고 앞단에 ALB (AWS Application Load Balancer)를 두어 요청을 여러 서버로 분산시킵니다. (Redis를 통해 상태를 공유해야 서버 1번과 2번이 동일한 정보를 제공할 수 있습니다).

## 요약 (Roadmap)
상용화 B2C 서비스 전환 시 달성해야 할 필수 마일스톤:
1.  **[네트워크]** 주기적 폴링(GET Request) 코드를 걷어내고 Socket.io 또는 FastAPI WebSocket으로 대체
2.  **[데이터베이스]** CSV 파일 의존도를 낮추고 Redis 캐시 시스템 구축
3.  **[네트워크]** Nginx 도입을 통한 정적 에셋 서빙 최적화 (Load Balancer 연계)
